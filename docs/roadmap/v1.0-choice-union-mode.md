# Version 1.0 Choice Union Mode

Plan for implementing full `xs:choice` union support behind `--client-choice-mode union`.

See the root [README.md](../../README.md) for project overview and [Version 1.0 Roadmap Plan](README.md) for the complete 1.0 route.

## Goal

`--client-choice-mode union` must generate TypeScript, OpenAPI, gateway schemas, and test artifacts that preserve `xs:choice` exclusivity. The existing `all-optional` mode must remain the default and must keep its current generated output unless a bug is explicitly fixed.

## Design Direction

The compiler must preserve choice-group structure in the catalog instead of flattening all choice children into ordinary optional elements. Downstream emitters can then decide how to render the same catalog in `all-optional` mode and `union` mode.

Union mode should model each choice branch as an exclusive object branch. For TypeScript, branch properties should exclude peer branches with `?: never` where that improves compile-time safety. For OpenAPI, the compatibility slice must decide whether `oneOf`, `anyOf`, or another constrained shape is safe for Fastify.

## Scope

- Preserve choice-group metadata during schema compilation.
- Emit TypeScript union types for choice groups in union mode.
- Keep default all-optional TypeScript output stable.
- Emit OpenAPI schemas that reflect choice exclusivity where Fastify-compatible.
- Generate mock data that selects a valid choice branch.
- Generate gateway tests that send one valid branch and reject invalid branch combinations when validation can enforce it.
- Update README, CLI reference, concepts, supported patterns, generated code docs, and migration docs.

## Out Of Scope

- Do not make union mode the default before 1.0.
- Do not change XML runtime mapping for non-choice elements.
- Do not solve abstract substitution groups in this slice unless required by choice fixtures.

## Implementation Units

### Catalog Metadata

Add catalog metadata for choice groups with stable names, branch element names, branch type names, occurrence constraints, nillable flags, and source order. The metadata must preserve nested choice groups well enough for downstream emitters to render deterministic output.

Initial progress in the `0.25.3` patch train starts with compiler metadata only. Emitters still use the flattened `elems` collection until the TypeScript and OpenAPI units consume `choiceGroups`.

### TypeScript Emitter

In `all-optional` mode, keep emitting the existing interface shape. In `union` mode, emit a base object for non-choice fields and combine it with a generated branch union.

The `0.25.3` patch train now emits TypeScript branch unions for `choiceGroups` when `--client-choice-mode union` is selected. The generated union uses peer `never` properties so TypeScript rejects payloads containing more than one branch.

### OpenAPI Emitter

Use the schema strategy proven by [OpenAPI Fastify Compatibility](v1.0-openapi-fastify-compatibility.md). The generated schema must reject multiple choice branches when the selected strategy can do so safely.

The selected request-body strategy is `oneOf` over containing-object branches with peer `not` constraints. Closed `oneOf` branches alone are not safe under Fastify defaults because AJV removes additional properties before evaluating the branch result.

The `0.25.3` patch train now emits that strategy for compiled `choiceGroups` in union mode. Choice member properties remain on the containing object, required non-choice properties stay top-level, and optional choice groups get an empty branch constraint.

### Gateway And Test Emitters

Generated gateway route schemas must consume the new OpenAPI shape without hand edits. Generated mock clients and test suites must produce valid branch data.

## Testing Approach

### Compiler Tests

Add fixtures for simple choice, optional choice, required choice, nested choice, choice mixed with attributes, and choice mixed with ordinary sequence elements. Assert catalog metadata before emitter work begins.

### TypeScript Tests

Add snapshot coverage for generated `types.ts` in both `all-optional` and `union` mode. Add compile-only tests that prove invalid multi-branch payloads fail when union mode is selected.

### OpenAPI Tests

Add unit or snapshot coverage for generated schema branches. Validate generated specs with the existing OpenAPI validation path.

### Gateway Tests

Add Fastify integration tests for valid choice request bodies and invalid multi-branch request bodies. Keep response serialization coverage conservative if Fastify compatibility research requires it.

### Regression Tests

Run existing snapshot tests to prove default output does not drift. Update snapshots only for intentional changes.

## Acceptance Criteria

- `--client-choice-mode union` changes generated output in a test-backed way.
- `all-optional` mode remains default and stable.
- TypeScript rejects invalid multi-branch payloads in union mode where the object model can represent exclusivity.
- OpenAPI schemas validate with the project validator.
- Generated Fastify routes accept valid branch payloads.
- Generated Fastify routes reject invalid branch combinations when the selected schema strategy supports that.
- Docs describe both modes and their tradeoffs accurately.

## Release Implications

This slice is a user-facing feature release. It should be called out as opt-in before 1.0, with no upgrade steps for users who keep the default mode.
